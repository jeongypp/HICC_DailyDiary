# 🤝 HICC_DailyDiary 협업 규칙입니다

## 1. 💬 커밋 메시지 규칙

커밋 메시지는 다음 형식을 지켜서 작성합니다.
`태그: 작업 내용 설명`

**[태그 종류]**
- `feat` : 새로운 기능 추가
- `fix` : 버그 및 오류, 오탈자 수정
- `docs` : 문서(.md) 추가 및 수정
- `refactor` : 코드 리팩토링 (기능 변경 없이 코드 구조 개선)
- `setting` : 레포지토리 환경 설정 등 기타 작업

**[작성 예시]**
- `feat: 일기 작성 페이지 UI 구현`
- `fix: Geminai API 호출 시 발생하는 오류 수정`

## 2. 🌿 브랜치 (Branch) 규칙

- `main` : 최종 브랜치 (반드시 팀 회의 후 develop에서 Merge, 직접 Push X)
- `develop` : 각 기능(feature)들을 하나로 합치는 통합 개발 브랜치 (각 feature에서 PR 후 여기에 Merge)
- `feature/기능명` : 기능별 파생 브랜치(각자 담당한 기능)
    - 예시: `feature/geminai-api`, `feature/diary-monthview`

## 3. 📢 이슈 (Issue) 규칙

- 효율적인 코드 평가와 오류 수정을 위해 '1 이슈 = 1 브랜치 = 1 PR'을 기본 원칙으로 합니다.
- 각자 맡은 독립적인 기능에 대해 먼저 이슈(계획)를 생성하고, 대응하는 브랜치를 파서 작업합니다.
- 버그 수정이나 코드 다듬기 등의 사소한 수정은 별도의 이슈나 브랜치를 생성하지 않아도 무방합니다.


**[작업 흐름 예시]**
1. 작업할 내용에 대한 이슈(Issue)를 먼저 등록.
2. `develop` 브랜치에서 파생하여 `feature/기능명` 브랜치 생성.
3. 기능 개발 완료하면 로컬에서 커밋 후 해당 브랜치에 Push.
4. 깃허브에서 `develop` 브랜치로 PR 남기기. (이때 PR 내용에 `Closed/Fixed #이슈번호`를 적으면 Merge시 이슈가 자동 종료됨.)
5. 팀원 간 확인 후 `develop` 브랜치에 Merge.
6. 오류 수정 등 모든 기능 개발이 끝난 `feature` 브랜치 삭제.

**[.gitignore 관련]**
개발 중 새로운 라이브러리나 환경 변수 파일(예: DB 비밀번호, API 키 등)이 추가되어 깃에서 무시해야 할 파일이 생기면, 공통 `.gitignore` 파일에 목록을 추가한 뒤 커밋 바랍니다.
